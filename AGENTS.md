## 2. 项目目标、读者与边界

### 2.1 目标读者

读者具备：

- 基础金融知识；
- Python、`pandas`、`numpy` 和基础数据分析能力；
- 但尚未独立完成过量化多因子研究。

### 2.2 主体结业目标

新人阅读讲义并完成实操后，应能够：

1. 独立完成一次基础的 A 股日频横截面因子研究；
2. 判断单因子是否具有进入简单多因子组合的增量价值；
3. 将综合信号转化为约束型多头组合；
4. 完成含成本回测；
5. 从预测力、风险、换手、成本和可交易性五个维度解释结果；
6. 明确结论边界、失效情形和未知口径。

### 2.3 主体非目标

除非用户明确扩展范围，主体不得深入展开：

- 高频、逐笔、盘口和订单簿策略；
- 期货、期权、可转债、跨资产和市场中性运营；
- 深度学习、强化学习、大模型直接选股；
- 复杂商业风险模型、完整二次规划器；
- 实盘交易系统、OMS/EMS、低延迟工程；
- 影子运行、上线审批、监控、回滚与模型退役的系统实现；
- 特定供应商字段级教程；
- 任何以回测收益最大化为目标的大规模参数搜索。

生产化、模型生命周期和进阶统计方法只能作为附录概览，不得打断主体教学主线。

## 5. 冻结的贯穿案例

案例名称：**宽基股票池三因子周频约束多头策略**。

### 5.1 基本设定

- 市场：A 股普通股票，日频横截面研究；
- 股票池：具有历史成分记录的、可通过配置替换的宽基股票池；
- 信号取样：每周最后一个交易日收盘后；
- 调仓：周频；
- 假设成交：下一交易日，以明确、可配置的日频价格代理；
- 组合：不加杠杆、不做空的约束型多头组合；
- 因子：中期动量、账面市值比价值、ROE 类质量；
- 合成：统一方向、完成预处理后等权平均；
- 组合构建：高分股票入选，使用规则型权重及行业、个股、流动性和换手约束；
- 验收：不要求正收益，不要求跑赢任何基准。

### 5.2 统一事件时序

```text
周度信号日 t 收盘后
  → 冻结 t 时点已可得数据
  → 计算因子与综合得分
  → 形成下一期目标权重
  → 下一交易日 t+1 按约定代理模拟成交
  → 持有至下一次调仓的假设成交时点
  → 记录实际持仓、收益、换手、成本和目标偏差
```

必须明确：

- 使用信号日收盘数据时，不得假设无条件以同一收盘价成交；
- 信号日股票池只能由信号日及以前的信息决定；
- 下一交易日的不可交易状态在成交环节处理，不得回头改写信号日样本；
- 所有实际成交价、涨跌停、停牌、复权、公司行动和历史费用口径均标记为“需实际确认”。

## 6. 研究正确性的不可违反约束

### 6.1 禁止虚构

严禁：

- 编造任何年化收益、IC、RankIC、夏普、回撤、胜率或产品表现；
- 用随机数生成看似真实的实证曲线，却未明确标注为“示意”；
- 虚构数据供应商、字段名、数据库表名或内部系统；
- 把示例参数写成行业统一口径；
- 把样本内结果表述为未来收益保证；
- 为了填满页面制造不必要的统计结果。

没有真实数据时：

- 使用概念表、空模板、极小玩具数据或明确标注的合成例子；
- 图表标题必须写明“示意”“结构模板”或“非实证结果”；
- 不得在合成例子中使用容易被误认为真实业绩的百分比曲线。

### 6.2 Point-in-Time 与时间边界

每个字段都应尽可能说明：

- 观察日期；
- 报告期；
- 公告日期；
- 数据可得日期；
- 生效日期；
- 修订版本；
- 信号截止时间。

财务数据不得按报告期末直接前填。若无法获得历史修订版本，必须披露限制并采用保守假设。

### 6.3 前视偏差与幸存者偏差

不得：

- 用当前指数成分回填历史；
- 只保留今天仍上市的股票；
- 用下一交易日是否停牌或涨跌停来删除信号日股票；
- 用全样本均值、方差、分位数或标准化参数处理早期样本；
- 让滚动窗口包含标签期数据；
- 用信号日收盘数据又以同一收盘价成交。

### 6.4 结论纪律

每个研究结论应区分：

- 观察到的事实；
- 基于事实的推断；
- 仍需验证的假设；
- 可执行建议；
- 局限与未知项。

允许结论为：

- 支持；
- 部分支持；
- 暂不支持；
- 证据不足。

不得因结果不漂亮而隐去失败实验。

正文中应清楚区分：

- **[通用原则]**：不依赖具体机构或供应商，通常应普遍遵守；

## 8. LaTeX 文档规范

### 8.1 文档类型

这是**连续阅读的 A4 技术讲义**，不是幻灯片。

必须：

- 使用 `ctexbook`、`ctexrep` 或仓库已确认的等价 A4 中文文档类；
- 使用 XeLaTeX；
- 保持 `main.tex` 为唯一主入口；
- 正文能够自然跨页；
- 章节、图、表、公式使用 `label` 与 `ref`/`cleveref`。

禁止：

- `beamer`；
- `\begin{frame}`；
- `\pause`、overlay、16:9 页面；
- 幻灯片式一页一个结论；
- 使用 `minted` 或依赖 `--shell-escape` 的方案；
- 依赖仓库外的私有字体或本地绝对路径；
- 依赖未提交的图片、数据或代码文件。

### 8.2 固定配色

若仓库中尚无 `preamble.tex`，使用以下颜色；若已存在同名颜色，不得随意改 RGB：

```latex
\definecolor{SoftBlue}{RGB}{76,112,150}
\definecolor{SoftGray}{RGB}{245,247,250}
\definecolor{BlockBlue}{RGB}{220,232,245}
\definecolor{BlockBlueDark}{RGB}{200,220,240}
\definecolor{BaseGray}{RGB}{135,145,158}
\definecolor{COPFBlue}{RGB}{60,102,170}
\definecolor{WarmOrange}{RGB}{210,126,71}
\definecolor{SoftGreen}{RGB}{83,145,126}
\definecolor{DarkText}{RGB}{28,31,35}
\definecolor{LightTan}{RGB}{247,243,236}
```

颜色语义：

- `SoftBlue`：章节标题、关键词、链接和流程箭头；
- `BlockBlue`：核心原则、定义和方法；
- `SoftGray`：代码背景、普通说明和章节概览；
- `WarmOrange`：常见错误、风险和待确认口径；
- `SoftGreen`：正确实践、检查清单和完成标准；
- `LightTan`：贯穿案例、直觉例子和教学情境；
- `BaseGray`：图注、页眉页脚和辅助标签；
- `DarkText`：正文。

页面应以白底深色正文为主，彩色面积保持克制。

### 8.3 排版与组件

- 表格使用 `booktabs`，避免密集竖线；
- 跨页表格使用 `longtable` 或等价方案；
- Python 代码使用 `listings`；
- 简单流程图、时序图优先使用 TikZ；
- 项目自定义 TikZ/PGF 样式键必须使用 `quant` 前缀；不得新增 `step`、`style`、`node`、`check`、`event`、`note`、`optional` 等过于通用的自定义键名；
- `every node/.style` 等标准 TikZ 键保持原名，不得为满足前缀规则而改写；
- 数据图如无真实数据，只画结构示意，不画虚构业绩；
- 外部图必须有来源、许可和相对路径；
- 图注应解释图的教学作用，而不是重复标题；
- 避免过长行、过小字号和超出版心的 TikZ 图；
- 公式只保留理解流程所需内容，不扩展成统计学教材；
- 中文正文使用完整段落，项目符号用于并列信息而非替代论证。

### 8.4 Overleaf 兼容性

- 不使用系统绝对路径；
- 不使用符号链接；
- 不要求联网下载包、字体或数据；
- 不使用需要本地守护进程的工具；

## 9. 技术写作规范

### 9.1 表达风格

- 默认使用中文；
- 专业但面向新人；
- 第一次出现的术语给出中文、英文和直觉解释；

- `references.bib` 中不得出现虚构 DOI、作者、期刊或网址；
- 不大段复制受版权保护的资料；引用应以转述为主。

## 10. Python、Notebook 与研究代码规范

### 10.1 基本原则

- 代码目标是清楚、可核对和可复现，不追求过度封装；
- 优先使用现有依赖与配置；没有环境说明时，不擅自增加大型依赖；
- 不在代码中写死供应商字段；使用概念层接口、配置映射或清晰占位；
- 不访问外部网络、不下载真实数据，除非任务明确允许；
- 不写入 API key、token、cookie、账号或绝对本地路径；
- 合成数据只用于单元测试和教学演示，并明确标记。

### 10.2 代码质量

- 公共函数使用类型标注和简短 docstring；
- 所有时间相关函数明确输入、输出和时区/交易日语义；
- 优先写小型纯函数；
- 关键逻辑必须有边界测试；
- 测试至少覆盖：
  - 公告日前后；
  - 周末和节假日；
  - 指数成分生效日前后；
  - 停牌或缺失成交价；
  - 标签起止边界；
  - 滚动窗口不足；
  - 分母为零或负权益；
  - 不可交易股票不会从持仓中凭空消失。

### 10.3 Notebook

- 必须从清空内核后顺序运行；
- 不依赖隐藏状态或手工改过的 CSV；
- 每组单元格包含目的、代码、输出和解释；
- 中间检查不能全部省略，只保留最终净值图；
- 输出数字必须由当前数据运行产生；无数据时使用空模板或结构示例；
- 重复函数可抽到 `src/`，但 Notebook 中保留关键输入、参数和结果解释。

建议记录字段：


要求：

## 13. Git 与 Overleaf 协作规范

### 13.1 Git 安全

禁止：

- `git reset --hard`；
- `git clean -fd`；
- 强制推送；
- 重写或 amend 用户已有提交；
- 擅自修改 Git remote；
- 擅自切换默认分支；
- 删除无法确认用途的文件；
- 覆盖未提交的用户修改。

要求：

- 默认继续当前分支，不创建新分支；
- 用户明确允许时，可以在完成检查后直接 commit 并 push 当前分支；
- 不 force push，不 amend 用户已有提交；
- 每个清晰阶段单独提交，提交信息准确说明结构、内容、样式或修复；
- push 前必须执行 `git diff --check`，并完成适用的编译或静态检查；
- 修改应小而聚焦；
- 提交前再次查看 `git diff` 和 `git status`；
- 任务结束时工作区应干净，或明确解释仍存在的未跟踪/未提交文件；
- 合并冲突时停止并报告，不凭猜测覆盖 Overleaf 或用户版本。

### 13.2 Overleaf 同步注意事项

GitHub 与 Overleaf 可能在不同时间写入同一分支，因此：

- 不进行大范围无意义格式化；
- 不在同一任务中同时改动所有章节；
- 不随意重命名主文件、章节文件或图片目录；
- 保持一次提交只解决一个清楚的问题；
- 不提交 LaTeX 临时产物；
- 若发现明显由 Overleaf 产生的并行修改，先报告状态，再继续；
- 不尝试修改 Overleaf 项目设置、编译器或外部集成配置，除非用户明确要求。

## 14. 验证与测试

### 14.1 LaTeX 编译

统一使用 XeLaTeX。若环境有 `latexmk`，优先运行并让其完成至少两轮所需的交叉引用收敛：

```bash
latexmk -xelatex -interaction=nonstopmode -halt-on-error main.tex
```

否则运行至少两次：

```bash
xelatex -interaction=nonstopmode -halt-on-error main.tex
xelatex -interaction=nonstopmode -halt-on-error main.tex
```

检查：

- 编译是否成功；
- 是否有缺失文件、未定义命令或未定义引用；
- 是否出现 `??` 交叉引用；
- 表格、代码和 TikZ 是否越出版心；
- 中文是否乱码；
- 目录和页码是否正确；
- 是否误用了 Beamer 命令。

普通 `overfull` 或 `underfull` box 警告不要求通过大范围改写正文消除；应记录真正影响可读性的版面问题，并保持本轮修改范围聚焦。

可使用静态检查：

```bash
rg -n '\\documentclass.*beamer|\\begin\{frame\}|\\pause|\\only|\\onslide' .
rg -n 'TODO|TBD|待补充|占位' chapters appendices frontmatter
```

占位章节必须清楚标记且不能假装已完成。

不得提交 `.aux`、`.log`、`.toc`、`.out`、`.fls`、`.fdb_latexmk`、`.synctex`、`.pdf` 等构建产物，除非用户明确要求版本化发布 PDF。

若环境没有 TeX：

- 不要声称“编译通过”；
- 完成静态检查；
- 在总结中明确写“未在当前环境编译验证”。

### 14.2 Python 与 Notebook

在存在测试时运行仓库已有命令。若无统一命令，可视项目情况运行：

```bash
python -m pytest -q
```

若没有真实数据：

- 运行合成 fixture 的单元测试；
- 不声称完成真实回测；
- 明确哪些检查尚待真实数据验证。

## 15. 任务结束时的汇报格式

最终回复应包含：

1. **完成内容**：按文件说明改动；
2. **验证结果**：列出实际执行过的编译、测试和静态检查；
3. **未验证事项**：例如缺少 TeX、数据或团队口径；
4. **研究口径待确认项**：只列真正影响实现的事项；
5. **下一步建议**：最多一项，避免把当前任务扩成新项目；
6. **Git 状态**：提交、分支和工作区状态。

不要说“全部完成”而不提供验证证据。

## 16. Definition of Done

一次内容或代码任务只有在以下条件满足时才算完成：

- 改动与当前任务范围一致；
- 没有覆盖用户已有工作；
- 没有虚构研究结果、字段或制度口径；
- 所有依赖具体团队的信息已标记为“需实际确认”；
- LaTeX 文件结构清晰且使用 A4 连续文档；
- 能编译则已编译，不能编译则已明确披露；
- 图、表、公式和代码均有教学目的；
- AI 使用记录已更新；
- `git diff` 已人工检查；
- 工作区干净，或未提交状态已明确说明；
- 最终回复包含修改与验证证据。
